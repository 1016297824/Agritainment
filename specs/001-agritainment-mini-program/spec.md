**项目名称**: 农家乐小程序
**版本**: V1.0
**日期**: 2026-05-05
**状态**: 已确认（brainstorming通过）

## 1. 项目概述

### 1.1 项目背景

为农家乐业务开发一款微信小程序，提供餐饮预约点餐、产品服务售卖、种植体验三大核心功能，实现线上线下一体化服务。

### 1.2 项目目标

- 为客户提供便捷的预约、点餐、购买、监控服务体验
- 为员工提供高效的扫码核销、服务管理工具
- 为管理员提供完善的后台管理能力

### 1.3 用户角色

| 角色 | 描述 | 数量上限 |
|------|------|----------|
| 客户 | 通过手机号注册，使用小程序各项服务 | 10,000名 |
| 员工 | 由管理员添加或授权，操作核销和管理功能 | 50名 |
| 管理员 | 总管理 + 副管理，高权限后台管理 | 3-5名 |

## 4. 餐饮功能

### 4.1 桌位预约

#### 客户视角

- 可查看后三天每日中午/夜晚的可预约桌位
- 选择时间和桌位后锁定，显示"已预订"状态
- 取消预约需在预约时间6小时前，超时计爽约次数
- 超过3次爽约自动加入黑名单

#### 员工视角

- 查看所有桌位预约情况
- 可帮客户手动取消预约（超时仍计爽约次数）
- 可将桌位手动置入"用餐"状态

#### 管理员视角

- 查看所有桌位预约情况
- 增加或删除可预约桌位

### 4.2 点餐

#### 客户视角

- 扫描桌上二维码进入点单页面
- 查看菜品信息并选择下单
- 下单后桌位进入"用餐"状态，可继续加菜
- 无需在线结账，线下付款

#### 员工视角

- 可为客户换桌（原桌位"闲置"，新桌位"用餐"）
- 可帮客户退菜
- 客户结账时手动结账，自动生成账目

#### 管理员视角

- 增加/删除桌位，生成对应二维码
- 增加/删除/编辑菜品（图片、简介）
- 设置每日菜品数量，库存不足显示"售罄"

## 5. 产品及服务售卖功能

### 5.1 产品类型

- **实体产品**: 农副产品等
- **服务产品**: 手工制作香囊体验、钓鱼等

*统一使用卡券形式，线下兑换*

### 5.2 产品管理（管理员）

- 增加/删除/编辑产品（图片、简介、金额）
- 实体产品设置每日兑换数量
- 生成卡券二维码及12位唯一标识码
- 二维码可下载用于制作线下卡券

### 5.3 会员功能

#### 客户

- 购买年费会员（按年结算）
- 会员购买后获得赠送服务产品
- 会员购买单品享受折扣

#### 管理员

- 查看客户会员状态
- 为客户赠送/修改会员
- 设置会员价格和赠送产品

### 5.4 线上预约

#### 客户

- 购买服务产品后以卡券形式存入卡包
- 选择后三天日期预约（需有对应卡券）
- 预约后卡券锁死
- 取消规则同餐饮预约，超时计爽约次数

#### 员工

- 查看每日预约情况
- 可帮客户取消预约（超时仍计次数）

#### 管理员

- 后台编辑每日每产品可预约数量

### 5.5 线下兑换

#### 客户

- 卡包内查看卡券，显示二维码和12位标识码

#### 员工

- 扫码验证卡券，验证后卡券进入"已用"状态
- 支持输入标识码验证

## 6. 种植体验功能（核心主打）

### 6.1 种植园规模

| 项目 | 规格 |
|------|------|
| 总面积 | 约100亩 |
| 每会员地块 | 约10平米 |
| 摄像头 | WiFi通信360°全景摄像头（**硬件预留接口**） |

### 6.2 客户功能

| 功能 | 说明 |
|------|------|
| 租用下单 | 线上购买，生成卡券（含二维码和12位标识码） |
| 实时监控 | 控制摄像头（放大、转动） |
| 管理服务 | 下单浇水、施肥等服务，通知园丁完成后提示 |

### 6.3 员工功能

| 功能 | 说明 |
|------|------|
| 核销卡券 | 扫描种植园服务卡券或输入标识码 |
| 地块绑定 | 核销后进入地块选择界面完成绑定 |
| 线下绑定 | 输入客户身份码为客户绑定地块 |
| 服务通知 | 接收客户管理服务下单通知，完成后点击完成 |
| 监控查看 | 可接入摄像头查看情况 |

### 6.4 管理员功能

| 功能 | 说明 |
|------|------|
| 地块管理 | 增加/删除地块，设置编号、名称、详情 |
| 摄像头管理 | 下线不需要的摄像头，添加新配置 |
| 绑定管理 | 摄像头与地块绑定（1:N，一个摄像头覆盖多个地块），客户租用后自动授权 |

### 6.5 摄像头冲突规则

| 场景 | 规则 |
|------|------|
| 同时操作 | 第一个客户锁定，第二个提示"使用中"，进入等待队列 |
| 等待队列 | 第三、第四个客户依次排队 |
| 超时处理 | 管理员设置最长使用时间，超时挤出当前客户 |
| 员工优先 | 员工和客户冲突时，员工插入队列第二位 |

## 8. 数据实体

### 8.1 核心实体

| 实体 | 主要属性 |
|------|----------|
| 用户 | 手机号、身份码、角色类型、会员状态、爽约次数 |
| 桌位 | 编号、状态（空闲/已预订/用餐）、二维码 |
| 菜品 | 名称、图片、简介、价格、库存数量 |
| 产品 | 类型(实体/服务)、名称、图片、简介、价格、库存/可预约数 |
| 卡券 | 标识码(12位)、类型、关联产品、状态(可用/已用/已锁定) |
| 预约 | 用户、桌位/产品、预约时间、状态 |
| 地块 | 编号、名称、详情、绑定摄像头列表、绑定客户 |
| 摄像头 | 编号、类型、IP地址、状态、绑定地块 |
| 朋友圈 | 标题、内容、图片、作者、发布时间、分享状态 |

### 8.2 实体关系

```
用户 ──(1:N)──> 卡券
用户 ──(1:1)──> 预约 (同一时段只能有一个预约)
用户 ──(1:1)──> 地块 (租用的)
摄像头 ──(1:N)──> 地块 (一个摄像头覆盖多个地块)
桌位 ──(1:1)──> 二维码
卡券 ──(N:1)──> 产品
```

## 10. 假设与约束

### 10.1 技术假设

- 前端使用 Uni-app + Vue 3 + Pinia 跨平台方案
- 后端使用 Spring Boot 3.2 + Java 21 + MyBatis-Plus
- 数据库使用 MySQL 8.0
- 认证使用 JWT (JJWT)
- 密码加密使用 BCrypt (jbcrypt)
- 图片存储使用服务器本地 + Nginx（后期可迁移COS）
- API使用RESTful风格

### 10.2 业务约束

- 第一期不接入微信支付，所有付款线下完成（预留支付接口）
- 摄像头设备需支持RTSP/Onvif协议（**第一期仅预留接口，硬件暂不接入**）
- 服务器需具备公网IP和域名 + HTTPS（Let's Encrypt免费证书）
- 前端代码上传微信服务器自动托管，后端部署到自有轻量服务器

### 10.3 架构决策（brainstorming确认）

| 决策 | 选择 | 理由 |
|------|------|------|
| 角色入口 | 单小程序 + 角色路由 | 维护成本低，保证微信小程序可用 |
| 支付 | 第一期无支付，预留接口 | 最小开发原则 |
| 部署 | 前端微信托管 + 后端轻量服务器 | 微信自动托管前端，后端自主控制 |
| 存储 | 服务器本地 + Nginx | 已有服务器资源充足，无需额外付费 |
| 摄像头 | 第一期仅预留接口 | 涉及硬件，先预留后接入 |

### 10.4 第一期范围（最小开发原则）

| 模块 | 第一期交付 | 后期 |
|------|-----------|------|
| 账户体系 | ✅ 客户注册、员工/管理员账户 | 微信一键登录 |
| 餐饮功能 | ✅ 桌位预约、点餐 | - |
| 产品售卖 | ✅ 卡券购买（线下付款）、会员管理、线下兑换 | 微信支付 |
| 种植体验 | ✅ 地块管理、卡券核销、摄像头绑定 | 摄像头实时监控 |
| 个人中心 | ✅ 卡包、朋友圈分享 | - |
| 支付 | ❌ 预留接口 | 微信支付 |
| 摄像头 | ❌ 预留接口 | PTZ控制+视频流 |

### 10.5 已知限制与待优化项

| 项目 | 状态 | 说明 |
|------|------|------|
| 图片存储 | 🚫 暂不处理 | 当前上传至服务器本地 uploads/，单机部署可用，水平扩展需接入 OSS/COS |
| 微信支付 | ⏳ 待实现 | 下单后无实际支付流程，订单直接创建，需接入微信支付 API |
| 消息通知 | ✅ 已实现 | 已接入微信订阅消息，支持预约成功/取消/服务下单通知。需在微信后台申请模板ID后填入配置 |
| 定时任务 | ✅ 已实现 | Spring @Scheduled 实现每日库存重置、预约超时标记爽约、会员过期检查。卡券过期自动处理未实现（卡券有 expired 状态但无定时任务触发），需后续补充 |
| Nginx 部署 | ⏳ 待实现 | 生产环境反向代理和静态文件服务配置 |
| TabBar 导航 | ✅ 已处理 | 使用微信原生 tabBar（5 个客户角色 tab + 图标）。自定义 TabBar 方案（CustomTabBar/AppTabBar 组件嵌入页面）因微信渲染层级问题不可用，已回退到原生方案。员工/管理员页面通过 uni.navigateTo/reLaunch 访问 |
