# Feature Specification: 餐饮模块功能增强

**Feature Branch**: `005-modify-catering`  
**Created**: 2026-05-11  
**Status**: Draft  
**Input**: 对餐饮模块进行功能修改，包括预约功能优化、点餐功能实现、页面结构调整

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 日期独立的桌位预约 (Priority: P1)

用户进入餐饮模块后，可以查看未来3天每个桌位在不同日期、不同时段（中午/夜晚）的独立预约状态。用户选择一个日期和时段后，看到该时段内每个桌位的可预约状态，选择一个空闲桌位完成预约。每个桌位在不同日期的预约相互独立——例如A桌在今天中午被预约，不影响A桌在明天中午或今晚的可预约状态。

**Why this priority**: 这是餐饮模块的核心基础功能。当前存在"一个桌位被预定则所有日期均显示被预定"的 bug，直接影响用户正常使用预约功能，必须优先修复。

**Independent Test**: 可以通过以下操作独立验证：预约A桌在今天中午 → 切换到明天中午 → 验证A桌仍显示为"可预约" → 切换到今晚 → 验证A桌仍显示为"可预约"。同时验证B桌在今天中午仍可预约。

**Acceptance Scenarios**:

1. **Given** 用户进入餐饮预约页面，选择"今天"和"中午"时段， **When** 用户预约A桌， **Then** A桌在今天中午显示"已预订"，且其他用户无法再预约A桌今天中午
2. **Given** A桌已在今天中午被预约， **When** 用户切换到"明天"和"中午"时段， **Then** A桌在明天中午显示为"可预约"
3. **Given** A桌已在今天中午被预约， **When** 用户切换到"今天"和"夜晚"时段， **Then** A桌在今晚显示为"可预约"
4. **Given** A桌已在今天中午被预约， **When** 另一个用户尝试预约A桌今天中午， **Then** 系统拒绝预约并提示"该桌位已被预约"
5. **Given** 用户已有一个当天中午的预约， **When** 用户尝试再次预约当天中午的另一个桌位， **Then** 系统提示"该时段已有预约"

---

### User Story 2 - 点餐功能实现 (Priority: P2)

用户在完成预约后，可以进入点餐页面浏览菜品菜单，将菜品加入购物车、调整数量，确认后提交订单。点餐页面需要显示用户已预约的桌位信息作为订单关联的依据。

**Why this priority**: 当前用户能看到菜品列表但无法点餐（点餐功能仅通过扫码方式可用），预约和点餐之间缺少业务闭环。这是餐饮体验的核心环节。

**Independent Test**: 可以通过以下操作独立验证：完成一个预约 → 进入点餐页面 → 浏览菜品列表 → 添加2份"宫保鸡丁"到购物车 → 调整数量为1份 → 查看购物车汇总 → 提交订单 → 验证订单创建成功并关联到预约桌位。

**Acceptance Scenarios**:

1. **Given** 用户已完成预约（桌位A，今天中午）， **When** 用户进入点餐页面， **Then** 页面顶部显示"已预约：A桌 - 今天中午"信息
2. **Given** 用户在点餐页面， **When** 用户浏览菜品列表， **Then** 显示所有可用的菜品及价格
3. **Given** 用户在点餐页面， **When** 用户点击某菜品的"+"按钮， **Then** 该菜品加入购物车，数量显示为1
4. **Given** 购物车中已有1份"宫保鸡丁"， **When** 用户再次点击"+"， **Then** 数量变为2
5. **Given** 购物车中有菜品， **When** 用户点击"-"按钮， **Then** 数量减1；减至0时从购物车移除
6. **Given** 购物车中有菜品， **When** 用户点击"下单"， **Then** 订单创建成功，显示订单确认信息
7. **Given** 用户未完成预约， **When** 用户尝试进入点餐页面， **Then** 系统提示"请先完成预约"并引导至预约页面
8. **Given** 某菜品库存为0， **When** 用户查看菜品列表， **Then** 该菜品显示"售罄"且无法加入购物车

---

### User Story 3 - 页面结构调整 (Priority: P2)

餐饮模块从单一页面拆分为两个独立页面：预约页面和点餐页面。两个页面之间有清晰的业务流程导航——用户必须先完成预约，系统自动引导至点餐页面；用户也可通过底部导航在预约和点餐之间切换。

**Why this priority**: 页面拆分是点餐功能实现的前提条件，也是优化用户体验的必要步骤。与 P2 点餐功能同等重要，两者配合实现完整的业务流程闭环。

**Independent Test**: 可以通过以下操作独立验证：进入餐饮模块 → 确认首先看到预约页面 → 完成预约后自动跳转到点餐页面 → 在点餐页面可以看到返回预约页面的入口 → 底部导航显示预约/点餐两个 tab。

**Acceptance Scenarios**:

1. **Given** 用户进入餐饮模块， **When** 页面加载完成， **Then** 默认显示预约页面，底部有"预约"和"点餐"两个导航项
2. **Given** 用户在预约页面完成预约， **When** 预约成功后， **Then** 系统提示"预约成功"并自动跳转到点餐页面
3. **Given** 用户在点餐页面， **When** 用户点击底部"预约"导航， **Then** 切换到预约页面，保留之前的预约信息
4. **Given** 用户未完成预约， **When** 用户点击底部"点餐"导航， **Then** 系统提示"请先完成预约"，不跳转
5. **Given** 用户有预约记录， **When** 用户进入点餐页面， **Then** 自动选中最近一次有效预约作为当前订单关联

---

### Edge Cases

- 用户预约后取消了预约，再进入点餐页面时应提示"请先完成预约"
- 用户在预约页面选择了日期和时段，但未提交就切换到点餐 tab，应提示而非静默丢失状态
- 同一天同一时段用户已有预约，尝试再预约另一个桌位时需正确拒绝
- 菜品库存为0时，购物车操作（+/-）应被禁用
- 用户下单时网络异常，应有明确的错误提示且购物车数据不丢失
- 预约日期已过（今天中午的预约在下午查看时），该预约应标记为过期
- 用户快速连续点击"预约"按钮，系统应防止重复提交
- 点餐页面加载时，如果用户没有有效预约，应展示空状态引导

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须按日期和时段独立管理每个桌位的预约状态，同一桌位在不同日期的预约互不影响
- **FR-002**: 系统必须在用户预约时校验目标桌位在指定日期和时段是否已被预约，已预约则拒绝并提示
- **FR-003**: 系统必须校验同一用户在相同时段内只有一个有效预约，防止重复预约
- **FR-004**: 系统必须将餐饮模块拆分为预约页面和点餐页面两个独立页面
- **FR-005**: 系统必须提供底部导航栏，包含"预约"和"点餐"两个入口
- **FR-006**: 系统必须在用户未完成有效预约时阻止进入点餐页面并给予明确提示
- **FR-007**: 系统必须在预约成功后自动跳转至点餐页面，并将预约的桌位信息传递给点餐页面
- **FR-008**: 点餐页面必须展示用户当前关联的预约桌位信息（桌号、日期、时段）
- **FR-009**: 点餐页面必须展示所有可用菜品列表，包括菜品名称、价格、描述和库存状态
- **FR-010**: 用户必须能够在点餐页面将菜品加入购物车、调整数量、移除菜品
- **FR-011**: 系统必须实时计算并显示购物车中菜品的总数量和总金额
- **FR-012**: 用户必须能够提交购物车内容生成订单，订单必须关联到用户的预约桌位
- **FR-013**: 系统必须在订单提交成功后清空购物车并显示订单确认信息
- **FR-014**: 预约页面必须保持与当前版本一致的日期选择（今天/明天/后天）和时段选择（中午/夜晚）功能
- **FR-015**: 系统必须在桌位列表中使用一致的视觉状态标识（可预约/已预订）
- **FR-016**: 两个页面的 UI 设计风格必须保持一致，使用相同的配色方案、字体和间距

### Key Entities

- **桌位预约 (TableReservation)**: 代表用户在特定日期和时段对桌位的预约。包含用户ID、桌位ID、预约日期、时段（中午/夜晚）、状态（pending/cancelled/checked_in）。核心变更：预约状态管理由按桌位改为按日期+时段+桌位三维管理。
- **桌位 (DiningTable)**: 代表餐厅中的物理桌位。包含桌号、容量、二维码。核心变更：桌位的 status 字段不再直接表示预约状态，预约状态由 TableReservation 按日期查询决定。
- **订单 (Order)**: 代表用户在某个桌位上的点餐订单。包含桌位ID、用户ID、总金额、状态（active/settled）。核心变更：订单需要关联到用户的预约记录而非仅依赖桌位二维码。
- **菜品 (Dish)**: 代表可点选的菜品。包含名称、价格、图片、描述、库存信息。
- **订单项 (OrderItem)**: 代表订单中的单个菜品项，包含菜品ID、数量、单价、状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 同一桌位在不同日期的预约状态完全独立，用户可以在3天×2个时段中的任意组合中独立预约每个桌位
- **SC-002**: 用户从进入预约页面到完成预约的操作步骤不超过4步（选日期→选时段→选桌位→确认）
- **SC-003**: 用户从进入点餐页面到提交订单的操作流程顺畅，无需额外的扫码或输入桌号操作
- **SC-004**: 预约状态的正确率达到100%，不会出现"预约已满"的误报或"可预约"的漏报
- **SC-005**: 点餐页面购物车操作响应即时（无感知延迟），数量变更和金额计算同步更新
- **SC-006**: 未预约用户点击点餐时的拦截率达到100%

## Assumptions

- 当前项目已有完整的用户认证系统，预约和点餐操作均需要用户登录
- 现有的菜品数据（dishes 表）和桌位数据（tables 表）结构基本满足需求，只需调整预约逻辑
- 当前已有的订单创建接口（POST /api/v1/dining/orders）可以复用，但需要调整请求参数（不再依赖桌位二维码，改为关联预约ID）
- 用户在一个时段内最多预约一个桌位（现有逻辑保持一致）
- 预约页面保持现有的3天日期范围（今天/明天/后天）和两个时段（中午/夜晚）
- 小程序平台（uni-app）的页面路由和底部导航机制可用于页面拆分
- 点餐页面的提交订单功能不需要支付环节（后续迭代添加）
