### 七、关键缺失项总结

#### 严重缺失（P1）

1. **无请求访问日志**：所有 HTTP 请求/响应无日志记录，无法追踪用户行为、排查接口问题、统计 QPS
2. **无日志文件输出**：生产环境日志仅输出到控制台，进程重启后日志丢失，无法事后排查
3. **无日志滚动策略**：即使配置了文件输出，也无大小/日期滚动，磁盘可能被撑满
4. **Interceptor 无日志**：AuthInterceptor 和 RoleInterceptor 在认证/鉴权失败时**不记录任何日志**，安全事件无审计痕迹

#### 重要缺失（P2）

5. **Controller 层零日志**：9 个 Controller 没有任何日志，核心业务操作（下单、预约、支付等）完全无记录
6. **Service 层大面积缺失**：9 个 Service 无日志，包括 AuthService（登录认证）、DiningService（订单）、CouponService（优惠券）等关键业务
7. **NotificationService 声明了 @Slf4j 但未使用**：异步通知发送成功/失败均无日志
8. **前端请求层零日志**：HTTP 请求/响应/错误完全不记录，前端问题排查只能靠用户截图

#### 一般缺失（P3）

9. **无 logback-spring.xml**：无法定制日志格式（如添加 traceId、用户ID）、无法配置不同包不同级别、无法配置日志归档策略
10. **无结构化日志**：所有日志都是纯文本，无法对接 ELK/Loki 等日志平台
11. **无请求追踪 ID**：无法将同一请求的多条日志串联
12. **前端无日志库**：无法按级别控制日志输出、无法上报错误到监控平台

## 方案对比

### 方案 A：AOP 切面 + HandlerInterceptor（推荐）

**思路**：用 Spring AOP 切面自动拦截 Controller/Service 方法记录业务日志，用 HandlerInterceptor 记录请求访问日志，用 Filter 记录请求耗时。

| 维度 | 评价 |
|------|------|
| **侵入性** | 低 — 业务代码只需加 `@Slf4j`，核心日志逻辑在切面中 |
| **一致性** | 高 — 切面统一控制日志格式、级别、脱敏规则 |
| **灵活性** | 中 — 特殊场景（如条件日志）需额外注解或手动 log |
| **维护成本** | 低 — 新增 Controller/Service 自动被切面覆盖 |
| **实现复杂度** | 中 — 需要写切面、拦截器、过滤器，但都是标准 Spring 模式 |

### 方案 B：纯手动 @Slf4j 逐方法加日志

**思路**：在每个 Service 方法中手动添加 `log.info/warn/error` 调用。

| 维度 | 评价 |
|------|------|
| **侵入性** | 高 — 每个方法都要改 |
| **一致性** | 低 — 依赖开发者记忆，容易遗漏或格式不统一 |
| **灵活性** | 高 — 每个方法完全自定义 |
| **维护成本** | 高 — 新方法需要记得加日志 |
| **实现复杂度** | 低 — 就是加 log 语句 |

### 方案 C：AOP 切面 + 手动关键日志混合

**思路**：AOP 切面负责请求访问日志和通用入口/出口日志，关键业务操作（支付、核销、转赠等）在 Service 中手动加详细日志。

| 维度 | 评价 |
|------|------|
| **侵入性** | 中 — 关键方法手动加，普通方法切面覆盖 |
| **一致性** | 高 — 请求日志统一，业务日志按需细化 |
| **灵活性** | 高 — 关键操作可自定义详细内容 |
| **维护成本** | 中 — 需要区分哪些手动加哪些自动覆盖 |
| **实现复杂度** | 中 |

---

**我推荐方案 A**，理由：

1. **项目规模适中**（9 Controller + 12 Service），AOP 切面能统一覆盖，不会过度设计
2. **一致性最好** — 日志格式、级别、脱敏规则在切面中统一控制，不会出现"这个方法忘了加日志"的情况
3. **请求访问日志用 Interceptor + Filter 实现**，这是 Spring Boot 最标准的做法
4. **对于需要特殊处理的场景**（如敏感数据脱敏、条件日志），可以通过自定义注解扩展

## 设计 Section 3：Layer 1 — 请求访问日志

### RequestLoggingFilter（Servlet Filter）

**职责**：记录每个 HTTP 请求的耗时和基本信息。

```
请求进入 → 记录 startTime 到 request attribute
请求完成 → 计算耗时，记录日志

日志格式：
[ACCESS] POST /api/v1/auth/login 200 45ms userId=3
[ACCESS] GET  /api/v1/dining/tables 200 12ms userId=3
[ACCESS] POST /api/v1/dining/reservations 201 89ms userId=3
[ACCESS] POST /api/v1/auth/login 401 5ms -
```

**关键规则**：
- 排除静态资源（`/uploads/**`）和健康检查
- 未登录用户 userId 显示为 `-`
- 响应状态码 4xx → WARN，5xx → ERROR，其余 → INFO
- 耗时超过 3 秒的请求额外标记 `[SLOW]`

### LoggingInterceptor（HandlerInterceptor）

**职责**：在 Handler 层补充请求上下文信息（userId、role），与 Filter 配合完成访问日志。

```
preHandle：从 request attribute 读取 userId/role，注入 MDC（Mapped Diagnostic Context）
afterCompletion：清理 MDC
```

**MDC 字段**：

| MDC Key | 来源 | 示例 |
|---------|------|------|
| `userId` | AuthInterceptor 注入的 request attribute | `3` |
| `role` | AuthInterceptor 注入的 request attribute | `CUSTOMER` |
| `requestId` | UUID 生成 | `a1b2c3d4` |

MDC 的好处：Logback 格式配置中可以直接引用 `%X{userId}`，所有日志行自动携带 userId，无需手动传参。

## 设计 Section 4：Layer 2 — 业务操作日志（AOP 切面）

### BusinessLogAspect

**职责**：通过 AOP 切面自动拦截 Service 层方法，记录业务操作日志。

**切点定义**：`com.agritainment.service..*Service.*(..)` — 拦截所有 Service 包下的方法。

### 日志策略

不是每个 Service 方法都记录入口/出口（那会产生大量噪音），而是按操作类型分级：

| 操作类型 | 日志方式 | 示例 |
|---------|---------|------|
| **写操作**（create/update/delete/cancel/settle/rent/purchase/grant/transfer/verify/refund/checkin/bind/share） | 记录入口 + 结果 | `[BIZ] createReservation userId=3, tableId=5, date=2026-05-08, timeSlot=lunch → reservationId=12` |
| **读操作**（get/list/query/find） | 不记录（避免噪音） | — |
| **异常** | 记录异常详情 | `[BIZ] createReservation FAILED userId=3, tableId=5 → 桌位已被占用` |

### 方法名匹配规则

```
写操作前缀（记录日志）：
  create*, save*, update*, delete*, remove*, cancel*, settle*,
  rent*, purchase*, grant*, transfer*, verify*, refund*, 
  checkin*, bind*, share*, unshare*, send*, control*, complete*

读操作前缀（跳过日志）：
  get*, find*, list*, query*, check*, is*, has*, count*
```

### 日志格式

```
成功：[BIZ] {methodName} {keyParams} → {resultSummary}
失败：[BIZ] {methodName} FAILED {keyParams} → {errorMessage}
```

### 参数脱敏

通过 `SensitiveDataUtils` 工具类处理：

| 参数类型 | 脱敏规则 | 示例 |
|---------|---------|------|
| phone | 中间4位替换为 `****` | `138****8899` |
| openid | 保留前4位和后4位 | `oX2b...k9Lm` |
| identityCode | 保留前4位和后4位 | `ABCD...WXYZ` |
| password | 完全隐藏 | `******` |
| code（验证码） | 完全隐藏 | `******` |

### 结果摘要

不记录完整返回对象（可能很大），只记录关键标识：

| 返回类型 | 记录内容 |
|---------|---------|
| Result 对象 | `code={code}, data.type={data.getClass().getSimpleName()}` |
| 实体对象 | `id={id}` |
| 集合 | `size={size}` |
| 基本类型 | 原值 |
| void | `ok` |

### IP 获取策略

```java
优先从 X-Forwarded-For 获取（反向代理场景）
其次从 X-Real-IP 获取
最后从 request.getRemoteAddr() 获取
X-Forwarded-For 取第一个 IP（原始客户端）
```

### 安全日志级别规则

| 事件 | 级别 | 理由 |
|------|------|------|
| Token 缺失/无效 | WARN | 可能是攻击探测 |
| Token 过期 | WARN | 可能是会话超时，也可能是 token 被篡改 |

**发现 1 个架构问题：**

**Issue #1: Filter 和 Interceptor 的执行顺序问题**

`RequestLoggingFilter` 是 Servlet Filter，在 Spring Interceptor 之前执行。但 `LoggingInterceptor` 在 Interceptor 层注入 MDC（userId, role）。这意味着：
- Filter `doFilter` 前：MDC 中没有 userId
- Filter `afterCompletion` 时：MDC 已被 LoggingInterceptor 清理

**解决方案**：在 `RequestLoggingFilter` 的 `afterCompletion` 中不依赖 MDC 获取 userId，而是从 request attribute 中读取（AuthInterceptor 已注入）。LoggingInterceptor 的 `afterCompletion` 只清理 requestId，不清理 userId/role（让 Filter 也能读到）。或者更简单：Filter 自己从 request attribute 读取 userId，不依赖 MDC。

**推荐**：RequestLoggingFilter 直接从 `HttpServletRequest.getAttribute("userId")` 读取 userId，不依赖 MDC。这样 Filter 和 Interceptor 完全解耦。

### Section 2: Code Quality Review

**Issue #2: AOP 切面参数序列化可能引发异常**

BusinessLogAspect 序列化方法参数时，如果参数对象没有合适的 `toString()`，可能抛异常或产生无意义的输出（如大对象的内存地址）。

**推荐**：使用安全的参数序列化策略 — 只记录参数名和简单类型值，复杂对象只记录类名+id（如果有的话）。

**Issue #3: 方法名前缀匹配的边界情况**

`checkNoShow` 以 `check` 开头，默认被归为读操作跳过，但它实际上会修改 no-show 计数。`incrementNoShow` 以 `increment` 开头，不在默认写操作列表中。

**推荐**：在 research.md 中已识别并规划了 `increment*` 前缀。`checkNoShow` 需要在切面中特殊处理（白名单方式，或改为 `markNoShow` 语义更清晰 — 但那是重构，不属于本次范围）。最简方案：将 `check*` 从读操作列表中移除，改为只匹配 `count*` 作为读操作。

# CEO 战略审查：业务日志系统计划

## 审查维度一：这是否是正确的问题？重新框定能否带来 10x 影响？

### 发现 1：问题框定过窄 — "日志系统" vs "可观测性"

**严重度：Critical**

计划将问题定义为"增加一个全面的业务日志系统"，但用户故事揭示的真实痛点是：
- 线上异常无法排查（可观测性）
- 安全事件无审计痕迹（合规性）
- 无法追溯用户操作历史（审计追踪）

这三个痛点的 10x 解法不是"更好的日志文件"，而是**可观测性体系**。当前计划花了大量精力在 AOP 切面、方法名前缀匹配、脱敏规则上，但这些只能解决"记录"问题，不能解决"发现"和"分析"问题。

**6 个月后什么会看起来很蠢？** 运维面对 1GB 的纯文本日志文件，用 grep 搜索 requestId，在多个归档文件间跳转，最终发现日志系统记录了一切但什么也找不到。

**修复建议**：重新框定为"可观测性基础设施"，至少应包含：
1. 结构化日志（JSON 格式），为未来对接 Loki/ELK 预留
2. 考虑 Spring Boot Actuator + Micrometer 提供指标监控
3. 安全审计事件应入库（数据库表），而非仅存文件

**严重度：Critical**

我逐一检查了所有 Service 方法，发现以下**关键写操作会被遗漏**：

**login 和 register 不被记录，这是安全审计的致命缺陷。** spec.md 的 User Story 2 明确要求"认证失败"需要审计，但登录成功的日志也不会被记录。

### 发现 4："单机部署，本地文件即可" 是未经验证的战略赌注

**严重度：High**

[spec.md](file:///e:/Workspace/AI/Agritainment/specs/003-business-logging/spec.md) Assumptions 第一条："项目为单机部署，日志写入本地文件即可满足需求，暂不需要对接 ELK/Loki 等日志平台"。

**6 个月后悔场景**：业务增长需要双机部署，发现两台机器的日志文件无法关联，requestId 在不同机器上不唯一（UUID 虽然几乎不碰撞，但跨机器 grep 是噩梦）。

**修复建议**：
1. 采用 JSON 结构化日志格式，即使当前只写本地文件，也为未来迁移预留
2. 在日志中加入 `hostname` 字段，为多机部署做准备
3. 明确写出"何时需要迁移到日志平台"的触发条件

### 发现 5："异步写入，耗时 < 1ms" 是空头承诺

**严重度：High**

[plan.md](file:///e:/Workspace/AI/Agritainment/specs/003-business-logging/plan.md) Technical Context 声称 "Performance Goals: 日志写入不影响业务请求响应时间（异步写入，耗时 < 1ms）"，但整个实现计划中**没有配置 AsyncAppender**。

Logback 默认的 `FileAppender` 和 `RollingFileAppender` 是**同步写入**的。每次日志调用都会阻塞业务线程直到磁盘写入完成。在磁盘 IO 繁忙时，日志写入可能耗时数十毫秒。

**修复建议**：在 `logback-spring.xml` 中配置 `AsyncAppender` 包装 `FileAppender`，并设置合理的 `queueCapacity` 和 `discardingThreshold`。

### 发现 6：MDC 在异步场景下的污染和丢失

**严重度：Medium**

计划在 `LoggingInterceptor.afterCompletion` 中清理 MDC，但未考虑：
1. **异步线程** — 如果 Service 中使用 `@Async` 或 `CompletableFuture`，子线程不会继承 MDC
2. **线程池复用** — Tomcat 线程池中，如果 `afterCompletion` 清理失败（异常场景），MDC 会污染下一个请求

**修复建议**：
1. 使用 `MDC.getCopyOfContextMap()` + `TaskDecorator` 传递 MDC 到异步线程
2. 在 Filter 层（比 Interceptor 更外层）做 MDC 清理，确保覆盖所有请求路径

## 审查维度四：哪些替代方案被轻率排除？

### 发现 9：自定义注解方案被错误否定

**严重度：High**

[research.md](file:///e:/Workspace/AI/Agritainment/specs/003-business-logging/research.md) 否定自定义注解的理由是"需要修改每个 Service 方法，侵入性高，且容易遗漏"。

但这个理由不成立：
1. **方法名前缀匹配同样容易遗漏** — 如发现 2 所示，`register`、`login`、`adminLogin`、`changeTable`、`staffCancelReservation` 全部遗漏
2. **注解是显式的，前缀匹配是隐式的** — 注解让意图清晰，新人看到 `@BusinessLog` 就知道这个方法有日志；前缀匹配是魔法行为，新人不知道哪些方法会被记录
3. **注解可以携带元数据** — 如操作描述、敏感参数标记、日志级别，比方法名推断更准确
4. **IDE 支持更好** — 可以搜索 `@BusinessLog` 引用，无法搜索"被 AOP 切面拦截的方法"

**修复建议**：重新评估注解方案。可以采用"默认记录 + 注解排除"的混合策略：
- 写操作默认记录（基于前缀匹配作为 fallback）
- 用 `@NoBusinessLog` 注解排除不需要日志的方法
- 用 `@BusinessLog(description="xxx")` 为关键方法提供更丰富的日志信息

### 发现 10：结构化 JSON 日志被轻率否定

**严重度：Medium**

否定理由："当前为单机部署，不需要对接日志平台，纯文本更易阅读"。

但 JSON 日志的优势不限于日志平台对接：
1. **可脚本化** — `cat agritainment.log | jq 'select(.userId==123)'` 比 grep 更精确
2. **字段可索引** — 不需要正则解析纯文本
3. **迁移零成本** — 未来对接 Loki/ELK 无需改格式
4. **脱敏更可靠** — 可以在序列化层面精确控制哪些字段脱敏，而非依赖参数名匹配

**修复建议**：至少在 logback-spring.xml 中配置一个 JSON 格式的 Appender 作为备选，通过 Spring Profile 切换。或者采用折中方案：人类可读格式 + 结构化 MDC 字段。

这些功能开箱即用，无需自己写 `RequestLoggingFilter`。

**修复建议**：评估 Spring Boot Actuator 是否可以替代 User Story 3 的自定义实现。如果可以，将精力集中在 User Story 1 和 2 上。

## 二、发现清单

### F-01: @Async 线程 MDC 上下文丢失

| 项目 | 内容 |
|------|------|
| **严重度** | Critical |
| **置信度** | 10/10 |
| **问题** | [NotificationService.java](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/service/NotificationService.java) 使用 `@Async` 注解，方法在 Spring 的异步线程池中执行。LoggingInterceptor 在请求线程的 `preHandle` 中注入 MDC，在 `afterCompletion` 中清理。但 `@Async` 方法运行在不同线程上，MDC 基于 `ThreadLocal` 实现，不会跨线程传播。结果是：异步方法中的日志（包括 BusinessLogAspect 如果拦截到的话）将丢失 requestId、userId、role 三个关键字段。 |
| **触发条件** | 任何触发通知的业务操作（创建预约、取消预约、创建服务订单）都会调用 `@Async` 方法。当前代码中 [DiningService.java:84](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/service/DiningService.java#L84)、[DiningService.java:106](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/service/DiningService.java#L106) 等处调用了 NotificationService。 |
| **影响** | 异步线程中的日志无法通过 requestId 串联回原始请求，运维排查时出现断裂。 |
| **修复** | 方案 A（推荐）：实现 `TaskDecorator`，在异步任务提交时复制 MDC 上下文到新线程。方案 B：在 LoggingInterceptor 中使用 `InheritableThreadLocal` 替代 MDC（不推荐，线程池复用会导致污染）。具体实现：创建 `MdcTaskDecorator` 类，配置到 `@EnableAsync` 的线程池中。 |

### F-02: Logback 未配置 AsyncAppender，同步 I/O 阻塞请求线程

| 项目 | 内容 |
|------|------|
| **严重度** | High |
| **置信度** | 9/10 |
| **问题** | plan.md 声称"异步写入，耗时 < 1ms"，但 research.md 和 plan.md 中均未配置 Logback 的 `AsyncAppender`。默认的 `FileAppender` 是同步写入磁盘的。在高并发场景下，每次日志写入都涉及磁盘 I/O，会直接阻塞请求线程。 |
| **触发条件** | 10x 负载下，假设每请求产生 3-5 条日志（访问日志 + 业务日志 + 可能的异常日志），每条日志同步写磁盘约 0.1-1ms，单请求额外增加 0.3-5ms 延迟。 |
| **影响** | 日志系统成为性能瓶颈，请求响应时间增加，与"日志写入不影响业务请求响应时间"的性能目标矛盾。 |
| **修复** | 在 `logback-spring.xml` 中用 `AsyncAppender` 包装 `FileAppender`，设置 `queueSize=256`、`discardingThreshold=0`（不丢弃日志）、`neverBlock=true`（避免日志阻塞业务线程）。注意：`neverBlock=true` 在队列满时会丢弃日志，需权衡。 |

### F-03: `code` 参数名脱敏规则过度匹配

| 项目 | 内容 |
|------|------|
| **严重度** | High |
| **置信度** | 9/10 |
| **问题** | [data-model.md](file:///e:/Workspace/AI/Agritainment/specs/003-business-logging/data-model.md) 规定参数名包含 `code` 时完全隐藏。但 `code` 是极其常见的英文词，会误匹配：`reservationCode`、`statusCode`、`errorCode`、`inviteCode`、`identityCode`（已有专门规则但也被 `*code*` 匹配）、`couponCode`、`qrCode` 等。这些参数对排查问题至关重要，完全隐藏会严重降低日志价值。 |
| **触发条件** | 任何包含 `code` 字段的 Service 方法参数。例如 [DiningService.java:116](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/service/DiningService.java#L116) 的 `getActiveOrder(String tableQr)` 如果参数名包含 `code`，会被脱敏。 |
| **影响** | 关键业务参数在日志中不可见，运维无法通过日志排查问题，违背日志系统的核心价值。 |
| **修复** | 将 `code` 的匹配规则改为精确匹配 `smsCode`/`verifyCode`/`authCode`，或者改为后缀匹配 `VerificationCode`/`ValidateCode`。不要用宽泛的 `*code*` 模式。同时建议脱敏规则改为白名单模式（只脱敏已知敏感字段），而非黑名单模式。 |

### F-04: AOP 切面与 @Transactional 的代理顺序问题

| 项目 | 内容 |
|------|------|
| **严重度** | High |
| **置信度** | 8/10 |

[JwtUtil.java](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/util/JwtUtil.java)

```java
package com.agritainment.util;

import com.agritainment.enums.RoleEnum;
import io.jsonwebtoken.Claims;
import io.jsonwebtoken.Jwts;
import io.jsonwebtoken.security.Keys;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;

import javax.crypto.SecretKey;
import java.nio.charset.StandardCharsets;
import java.util.Date;

@Component
public class JwtUtil {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.expiration}")
    private long expiration;

    private SecretKey getKey() {
        return Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    }

    public String generateToken(Long userId, String phone, RoleEnum role, boolean isMember) {
        return Jwts.builder()
                .subject(userId.toString())
                .claim("phone", phone)
                .claim("role", role.getValue())
                .claim("isMember", isMember)
                .issuedAt(new Date())
                .expiration(new Date(System.currentTimeMillis() + expiration))
                .signWith(getKey())
                .compact();
    }

    public Claims parseToken(String token) {
        return Jwts.parser()
                .verifyWith(getKey())
                .build()
                .parseSignedClaims(token)
                .getPayload();
    }

    public Long getUserId(String token) {
        return Long.parseLong(parseToken(token).getSubject());
    }

    public String getRole(String token) {
        return parseToken(token).get("role", String.class);
    }
}
```

## parseToken 方法抛出的异常类型分析

项目使用的是 **JJWT 0.12.5**（见 [pom.xml](file:///e:/Workspace/AI/Agritainment/server/pom.xml#L22)），`parseToken` 内部调用的是 `Jwts.parser().verifyWith(key).build().parseSignedClaims(token)`。

在 JJWT 0.12.x 中，`parseSignedClaims()` 会抛出以下异常类型（全部继承自 `io.jsonwebtoken.JwtException`，而 `JwtException` 继承自 `java.security.SignatureException`）：

| 异常类 | 触发场景 | 所属包 |
|---|---|---|
| **`ExpiredJwtException`** | Token 已过期（exp 声明早于当前时间） | `io.jsonwebtoken` |
| **`MalformedJwtException`** | Token 格式不合法（不是合法的 JWT 三段式结构、Base64 解码失败等） | `io.jsonwebtoken` |
| **`SignatureException`** | 签名验证失败（密钥不匹配、签名被篡改） | `io.jsonwebtoken.security` |
| **`UnsupportedJwtException`** | Token 的算法/格式不被当前 parser 配置支持 | `io.jsonwebtoken` |
| **`IncorrectClaimException`** | 声明值不符合预期（如 `verifyWith` 校验的 claim 不匹配） | `io.jsonwebtoken` |
| **`MissingClaimException`** | 缺少必需的声明 | `io.jsonwebtoken` |
| **`PrematureJwtException`** | Token 尚未生效（nbf 声明晚于当前时间） | `io.jsonwebtoken` |

**最常遇到的三种异常**（也是你重构 AuthInterceptor 最需要区分的）：

1. **`ExpiredJwtException`** — Token 过期，用户需要重新登录
2. **`MalformedJwtException`** — Token 格式错误，可能是伪造或损坏
3. **`SignatureException`** — 签名不匹配，密钥错误或 Token 被篡改

## 重构建议：区分异常类型

重构时可以按以下方式分层 catch，给前端返回不同的错误码和提示：

```java
try {
    String token = authHeader.substring(7);
    var claims = jwtUtil.parseToken(token);
    request.setAttribute("userId", Long.parseLong(claims.getSubject()));
    request.setAttribute("role", RoleEnum.fromValue(claims.get("role", String.class)));
    request.setAttribute("isMember", claims.get("isMember", Boolean.class));
    return true;
} catch (ExpiredJwtException e) {
    // Token 过期 — 可提示用户刷新 Token 或重新登录
    sendError(response, 40102, "Token已过期，请重新登录");
    return false;
} catch (MalformedJwtException e) {
    // Token 格式错误 — 可能是伪造
    sendError(response, 40103, "Token格式错误");
    return false;
} catch (SignatureException e) {
    // 签名验证失败 — Token 被篡改或密钥不匹配
    sendError(response, 40104, "Token签名无效");
    return false;
} catch (JwtException e) {
    // 其他 JWT 异常的兜底
    sendError(response, 40101, "Token无效");
    return false;
}
```

需要引入的 import：

```java
import io.jsonwebtoken.ExpiredJwtException;
import io.jsonwebtoken.JwtException;
import io.jsonwebtoken.MalformedJwtException;
import io.jsonwebtoken.security.SignatureException;
```

错误码设计建议：`40101` = 通用无效、`40102` = 过期、`40103` = 格式错误、`40104` = 签名无效，前端可根据不同错误码做差异化处理（如过期时自动跳转登录页，签名无效时提示安全风险）。

**Issue #1: BusinessLogAspect 双 @Around 导致重复拦截**

计划中 BusinessLogAspect 有两个 `@Around` advice：
1. `@Around("@annotation(com.agritainment.annotation.BusinessLog)")` — 拦截所有 @BusinessLog 注解方法
2. `@Around("execution(* com.agritainment.service..*Service.*(..))")` — 拦截所有 Service 方法

当一个方法同时有 @BusinessLog 注解且方法名以写操作前缀开头时，两个 advice 都会触发。虽然 `logServiceWriteMethods` 中有 `sig.getMethod().getAnnotation(BusinessLog.class) != null` 的检查来跳过，但这依赖 Spring AOP 的代理机制——如果同一个类内部调用（self-invocation），@BusinessLog 的 advice 可能不会触发，而 execution 的 advice 会触发。

更重要的是，**两个 @Around 在同一个 @Aspect 类中会导致不确定的执行顺序**，Spring AOP 不保证同一 aspect 内多个 advice 的执行顺序。

好，合并为单个 advice。我会在计划中修正 BusinessLogAspect 为单个 `@Around("execution(* com.agritainment.service..*Service.*(..))")`，在方法体内统一判断是注解方法还是前缀匹配方法。

**Issue #2: RequestLoggingFilter 与 LoggingInterceptor 的 MDC 时序问题**

RequestLoggingFilter 是 Servlet Filter（在 Interceptor 之前执行），而 LoggingInterceptor 负责注入 MDC requestId。Filter 的 `doFilterInternal` 在 Interceptor 的 `preHandle` 之前执行，所以 Filter 中读取 `request.getAttribute("requestId")` 时可能为 null（因为 requestId 还没被设置）。

当前代码中 RequestLoggingFilter 在 `finally` 块中读取 requestId，此时 Interceptor 已经执行过 `preHandle`，所以 requestId 应该已经存在。但 Filter 的 `finally` 块在 Interceptor 的 `afterCompletion` 之后执行吗？不——Filter 是外层包裹，执行顺序是：

```
Filter.doFilterInternal START
  → Interceptor.preHandle (设置 MDC requestId)
  → Controller
  → Interceptor.afterCompletion (清除 MDC)
Filter.doFilterInternal FINALLY (读取 requestId) ← MDC 已被清除！
```

所以 `request.getAttribute("requestId")` 能读到（存在 request attribute 中），但 MDC 中的 requestId 已被 afterCompletion 清除。不过当前代码是从 `request.getAttribute` 读取而非 MDC，所以没问题。

但有一个更微妙的问题：**Filter 在 Interceptor 之前注册，但 Interceptor 的 afterCompletion 在 Filter 的 finally 之前执行**，导致 MDC 在 Filter finally 中已被清除。如果 Filter 需要从 MDC 读取任何值，都会失败。当前代码只从 request attribute 读取，所以 OK，但这是一个脆弱的设计。

**Issue #3: AuthInterceptor 中 MDC userId/role 注入位置与 LoggingInterceptor 清理冲突**

Spring MVC 的行为是：当 Interceptor 的 `preHandle` 返回 false 时，后续 Interceptor 不会执行，但**前面已经返回 true 的 Interceptor 的 afterCompletion 会执行**。由于 LoggingInterceptor(order=1) 在 AuthInterceptor(order=2) 之前，所以 LoggingInterceptor.afterCompletion 一定会执行。这是正确的。

### C-1: IpUtils 盲信 X-Forwarded-For 头 -- IP 伪造/注入风险

[IpUtils.java:10-13](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/common/IpUtils.java#L10-L13)

```java
String ip = request.getHeader("X-Forwarded-For");
if (ip == null || ip.isEmpty()) ip = request.getHeader("X-Real-IP");
if (ip == null || ip.isEmpty()) ip = request.getRemoteAddr();
return ip;
```

**问题：**
1. `X-Forwarded-For` 头可包含多个 IP（`client, proxy1, proxy2`），代码直接取整个值而非解析第一个（真实客户端）IP。
2. 攻击者可伪造该头注入任意字符串，该值直接写入 `security_audit_log` 表的 `ip` 字段（VARCHAR(50)），超长值会被截断但恶意内容可存入数据库。
3. 无任何格式校验，攻击者可注入 `<script>` 等内容，若管理后台展示该数据则存在存储型 XSS 风险。
4. IP 伪造可绕过任何基于 IP 的限流或封禁逻辑。

**修复建议：**
```java
public static String getClientIp(HttpServletRequest request) {
    String xff = request.getHeader("X-Forwarded-For");
    if (xff != null && !xff.isEmpty()) {
        String ip = xff.split(",")[0].trim();
        if (isValidIp(ip)) return ip;
    }
    String realIp = request.getHeader("X-Real-IP");
    if (realIp != null && !realIp.isEmpty() && isValidIp(realIp)) return realIp;
    return request.getRemoteAddr();
}

private static boolean isValidIp(String ip) {
    return ip != null && ip.length() <= 45 && ip.matches("[0-9a-fA-F.:]+");
}
```

### C-4: BusinessLogAspect 对复杂对象 toString() 泄露敏感数据

[BusinessLogAspect.java:77](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/logging/BusinessLogAspect.java#L77)


当参数为实体对象（如 `User`、`Coupon`）时，Lombok `@Data` 生成的 `toString()` 会输出所有字段。例如 `CouponService.transfer()` 的 `couponId` 参数是 `Long` 类型没问题，但如果未来有方法接收 `Coupon` 对象，则 `toString()` 会暴露 `code`（couponCode）和 `qrCodeData`，违反 FR-008（couponCode/qrCode 不应脱敏但也不应在业务日志中完整输出）。

```java
private String maskParams(String[] paramNames, Object[] args) {
    if (paramNames == null || args == null) return "{}";
    StringBuilder sb = new StringBuilder("{");
    for (int i = 0; i < paramNames.length; i++) {
        if (i > 0) sb.append(", ");
        sb.append(paramNames[i]).append("=");
        if (args[i] == null) {
            sb.append("null");
        } else if (args[i] instanceof String s) {
            sb.append(SensitiveDataUtils.maskParam(paramNames[i], s));
        } else if (args[i] instanceof Number || args[i] instanceof Boolean) {
            sb.append(args[i]);
        } else {
            sb.append(args[i].getClass().getSimpleName()).append("@").append(Integer.toHexString(args[i].hashCode()));
        }
    }
    sb.append("}");
    return SensitiveDataUtils.truncate(sb.toString());
}
```

### I-2: BusinessLogAspect @Order(LOWEST_PRECEDENCE) 并不保证"事务提交后"日志

[BusinessLogAspect.java:29-31](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/logging/BusinessLogAspect.java#L29-L31)

`LOWEST_PRECEDENCE` 使 BusinessLogAspect 成为最内层通知（最靠近方法体），执行顺序为：

```
TransactionInterceptor 进入 → 开启事务
  BusinessLogAspect 进入 → 记录参数
    方法执行
  BusinessLogAspect 退出 → 记录 [BIZ_OK]  ← 仍在事务内！
TransactionInterceptor 退出 → 提交事务
```

若方法成功返回但事务提交失败（如乐观锁冲突），日志已记录 `[BIZ_OK]`，但数据实际未持久化。这与 Critical Gap #5 的意图（"事务提交后再记录"）不符。

**修复建议：** 若需严格保证"事务提交后"语义，应使用 `TransactionSynchronizationManager.registerSynchronization()` 或 `@TransactionalEventListener(phase = AFTER_COMMIT)`。当前方案作为折中可接受，但应在代码注释中明确此限制。

### I-5: AsyncAppender 高负载下会阻塞请求线程

[logback-spring.xml:64-86](file:///e:/Workspace/AI/Agritainment/server/src/main/resources/logback-spring.xml#L64-L86)

```xml
<appender name="ASYNC_ACCESS" class="ch.qos.logback.classic.AsyncAppender">

    <discardingThreshold>0</discardingThreshold>
```

`discardingThreshold=0` 意味着队列满时不丢弃任何级别的日志，且 `neverBlock` 默认为 `false`。当队列满时（512 条未消费），调用线程（即请求处理线程）将被阻塞等待队列有空位。在高并发场景下，日志写入可能成为性能瓶颈。

```xml
    <neverBlock>true</neverBlock>
    <appender-ref ref="ACCESS_FILE"/>
</appender>

<appender name="ASYNC_SECURITY" class="ch.qos.logback.classic.AsyncAppender">
```

## Finding 1: IP Spoofing via X-Forwarded-For — Attacker Controls Audit Log IP

**File:** [IpUtils.java:12-16](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/common/IpUtils.java#L12-L16)

`getClientIp()` blindly trusts the first value in `X-Forwarded-For`. Any client can set this header to any valid IP. The IP is then stored in `security_audit_log` and used in security event logging (`TOKEN_MISSING`, `TOKEN_EXPIRED`, `ROLE_DENIED`, etc.). An attacker can:

- Spoof their IP to frame another user in audit logs
- Evade IP-based rate limiting or blocking (if ever implemented)
- Poison audit trail for forensic investigations

The `isValidIp()` check only validates format, not provenance. There is no trusted proxy configuration — the code has no way to know how many reverse proxies sit in front, so it cannot know which `X-Forwarded-For` entry is trustworthy.

**Classification: FIXABLE** — Configure a trusted proxy count and walk the `X-Forwarded-For` chain from the right (or use Spring's `ForwardedHeaderFilter` with a configured trusted proxy count).

## Finding 8: MDC Leak Between Requests on Async Threads

**File:** [MdcTaskDecorator.java:13-25](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/config/MdcTaskDecorator.java#L13-L25) and [RequestLoggingFilter.java:62-64](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/logging/RequestLoggingFilter.java#L62-L64)

The `MdcTaskDecorator` copies MDC from the submitting thread to the async thread and clears it in `finally`. However, the `RequestLoggingFilter` clears MDC keys (`requestId`, `userId`, `role`) in its `finally` block on the request thread. If the async task starts after the request thread has already cleared MDC (race condition on slow async dispatch), the async thread gets an empty context map. But more subtly: if two async tasks are submitted from the same request thread, and the first task's `MDC.clear()` in the decorator runs while the second task is being decorated, the second task may get a stale or empty context.

The real problem: `MDC.clear()` in the decorator's `finally` block is aggressive — it clears ALL MDC keys, not just the ones it set. If the async thread had any pre-existing MDC context (e.g., from a previous task on a pooled thread that wasn't properly cleaned), it gets wiped. But the opposite is also true: if `MDC.clear()` fails or is skipped (exception before the `try`), MDC from one user's request leaks into another user's async task.

**Classification: INVESTIGATE** — Verify whether the `MDC.setContextMap(contextMap)` correctly restores the original context rather than just overlaying. The current implementation does not save/restore the async thread's prior MDC state.

## Finding 17: `BusinessLogAspect` Order Is `LOWEST_PRECEDENCE` — Runs After All Other Aspects Including Transactions

**File:** [BusinessLogAspect.java:29-31](file:///e:/Workspace/AI/Agritainment/server/src/main/java/com/agritainment/logging/BusinessLogAspect.java#L29-L31)

```java
@Override
public int getOrder() {
    return Ordered.LOWEST_PRECEDENCE;
}
```

`LOWEST_PRECEDENCE` means this aspect runs outermost (last in the interceptor chain). This means the `@Around` advice wraps around the `@Transactional` advice. So the business log records `[BIZ_OK]` BEFORE the transaction commits. If the transaction fails to commit (e.g., constraint violation at flush time), the log says success but the data was never persisted. This creates a false audit trail.

**Classification: FIXABLE** — Set the aspect order to `Ordered.HIGHEST_PRECEDENCE + 1` or similar, so it runs inside the transaction boundary. The log should only record success after the transaction actually commits.
