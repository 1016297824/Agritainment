## 1. using-superpowers — 使用 Superpowers

### 功能描述

**主要作用：** 定义如何在任何对话中查找和使用 Superpowers 技能。这是整个技能体系的入口规则，规定了技能调用的强制性要求。

**适用场景：**
- 每次新对话开始时
- 任何可能需要技能的回应之前
- 作为所有其他技能的元规则

**核心特性：**
- 强制技能检查规则：即使只有 1% 的可能性适用，也必须调用技能
- 技能优先级体系：流程技能 > 实现技能
- 技能类型分类：刚性技能（严格遵循）vs 弹性技能（灵活适配）
- 防合理化机制：内置红牌警告表，阻止跳过技能的借口
- 平台适配：支持 Claude Code、Copilot CLI、Gemini CLI 等不同环境

### 完整工作流程

**步骤 1：识别是否需要技能**
- 每次收到用户消息后，检查是否有任何技能可能适用于当前任务
- 标准：即使只有 1% 的可能性，也必须调用 Skill 工具

**步骤 2：区分技能优先级**
- 流程技能优先（如 brainstorming、debugging）—— 这些决定如何处理任务
- 实现技能其次（如 frontend-design、mcp-builder）—— 这些指导执行

**步骤 3：调用技能**
- Claude Code 环境：使用 `Skill` 工具
- Copilot CLI 环境：使用 `skill` 工具
- Gemini CLI 环境：使用 `activate_skill` 工具
- 不要使用 Read 工具读取技能文件

**步骤 4：宣布技能使用**
- 格式：`"Using [skill] to [purpose]"`

**步骤 5：遵循技能内容**
- 如果有 checklist，为每项创建 TodoWrite 任务
- 严格遵循技能指令执行

**输出结果：** 技能内容被加载并呈现，Agent 遵循技能定义的流程执行后续操作。

## 2. brainstorming — 头脑风暴

### 功能描述

**主要作用：** 将创意转化为完整的设计和规格说明，通过自然的协作对话精炼想法，最终产出可执行的设计文档。

**适用场景：**
- 创建新功能、构建组件、添加功能、修改行为之前（强制使用）
- 任何创意工作之前
- 新项目构思阶段

**核心特性：**
- 硬性门禁：在设计获得用户批准前，禁止调用任何实现技能、编写代码
- 一次一个问题：逐步精炼需求，避免信息过载
- 2-3 种方案对比：提供多方案比较，含推荐和权衡分析
- 增量验证：逐段呈现设计，每段用户确认后再继续
- YAGNI 原则：无情移除不必要的功能
- 可视化伴侣：支持浏览器端展示线框图、图表等视觉内容
- 自审查机制：设计文档完成后自动检查占位符、一致性、范围、歧义

### 完整工作流程

**步骤 1：探索项目上下文**
- 检查当前项目文件、文档、最近提交
- 评估范围：如果涉及多个独立子系统，先拆分再讨论

**步骤 2：提供可视化伴侣（条件性）**
- 当预期有视觉问题时，以独立消息询问用户是否启用浏览器可视化
- 启用后，每个问题判断用浏览器还是终端展示

**步骤 3：询问明确问题**
- 一次一个问题，逐步理解目的、约束、成功标准
- 优先多选题，开放性问题也可
- 识别是否需要拆分为子项目

**步骤 4：提出 2-3 种方案**
- 对话式呈现，附带权衡分析和推荐理由
- 推荐方案前置说明原因

**步骤 5：呈现设计**
- 覆盖：架构、组件、数据流、错误处理、测试
- 每部分按复杂度缩放（简单部分几句话，复杂部分 200-300 词）
- 每部分后检查用户是否认同

**步骤 6：编写设计文档**
- 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
- 提交到 git

**步骤 7：设计自审查**
- 占位符扫描：检查 TBD/TODO/不完整部分
- 内部一致性：各部分是否矛盾
- 范围检查：是否适合单个实现计划
- 歧义检查：是否有双解需求

**步骤 8：用户审查门禁**
- 用户审查 design doc 后才能继续
- 如需修改，修改后重新审查

**步骤 9：过渡到实现**
- 调用 writing-plans 技能创建实现计划
- 禁止调用其他技能

## 3. systematic-debugging — 系统化调试

### 功能描述

**主要作用：** 在任何修复之前找到根因。随机修复浪费时间并产生新 bug，快速补丁掩盖潜在问题。核心原则是"无根因调查，不提出修复"。

**适用场景：**
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**核心特性：**
- 铁律：修复前必须先做根因调查
- 四阶段流程：根因调查 → 模式分析 → 假设与测试 → 实现
- 多组件系统诊断：逐层添加日志定位故障组件
- 数据流追踪：从错误点反向追踪到原始触发点
- 科学方法：假说 → 最小测试 → 验证 → 推进
- 架构质疑机制：3+ 次修复失败后质疑架构合理性
- 防合理化表：覆盖 8 种常见跳过借口
- 辅助技术：root-cause-tracing.md、defense-in-depth.md、condition-based-waiting.md

### 完整工作流程

**Phase 1：根因调查**

1. **仔细阅读错误信息** — 不跳过任何错误或警告，完整阅读堆栈跟踪
2. **可靠复现** — 确定触发步骤、是否每次发生
3. **检查最近变更** — git diff、最近提交、新依赖、配置变更、环境差异
4. **多组件系统收集证据** — 在每个组件边界添加诊断日志
5. **追踪数据流** — 从错误深处反向追踪，找到错误值的起源

**Phase 2：模式分析**

1. **找到工作示例** — 在代码库中定位类似但正常工作的代码
2. **与参考对比** — 完整阅读参考实现（不略读）
3. **识别差异** — 列出工作和损坏之间的每个差异
4. **理解依赖** — 需要哪些其他组件、配置、环境、假设

**Phase 3：假设与测试**

1. **形成单一假设** — 明确表述："我认为 X 是根因，因为 Y"
2. **最小测试** — 做最小改动来测试假设，一次一个变量
3. **验证后继续** — 有效 → Phase 4；无效 → 形成新假设
4. **不知道时** — 直接说"我不理解 X"，不假装知道

**Phase 4：实现**

1. **创建失败测试用例** — 最简单的复现，使用 TDD 技能
2. **实现单一修复** — 只修复根因，一次一个改动，不做附带优化
3. **验证修复** — 测试通过？其他测试没坏？问题实际解决？
4. **修复无效时** — 少于 3 次失败：返回 Phase 1；3 次以上：质疑架构
5. **3+ 次修复失败** — 讨论是否有架构问题，不继续修复

**输出结果：** 根因定位报告包含：根因描述、触发条件、影响范围、修复方案、副作用评估。

## 4. test-driven-development — 测试驱动开发

### 功能描述

**主要作用：** 先写测试，看着它失败，然后写最小代码让它通过。核心原则是"没有看到测试失败，你不知道它是否测试了正确的东西"。

**适用场景：**
- 新功能开发
- bug 修复
- 重构
- 行为变更

**核心特性：**
- 铁律：没有先失败的测试，就不能有生产代码
- RED-GREEN-REFACTOR 循环：红色（写失败测试）→ 绿色（最小代码）→ 重构（清理）
- 双验证机制：红阶段验证测试正确失败，绿阶段验证测试通过
- 严格删除规则：在测试之前写的代码必须删除，不能保留作参考
- 测试质量标准：一个行为、清晰命名、真实代码（最少 mock）
- 测试反模式指南：避免测试 mock 行为、避免给生产类添加测试专用方法
- 防合理化表：覆盖 11 种常见跳过借口
- 调试集成：发现 bug → 写失败测试复现 → TDD 周期 → 修复证明

### 完整工作流程

**RED 阶段 — 编写失败测试**

1. 编写一个最小测试，展示应该发生什么
2. 要求：一个行为、清晰名称、真实代码

**验证 RED — 确认测试正确失败（强制）**

1. 运行测试命令
2. 确认：测试失败（非错误）、失败消息符合预期、因功能缺失失败（非拼写错误）
3. 测试通过？说明测试的是已有行为，需修正测试

**GREEN 阶段 — 编写最小代码**

1. 写最简单的代码让测试通过
2. 只够通过测试即可，不添加功能、不重构其他代码

**验证 GREEN — 确认测试通过（强制）**

1. 运行测试命令
2. 确认：测试通过、其他测试仍然通过、输出干净
3. 测试失败？修代码；其他测试失败？立即修复

**REFACTOR 阶段 — 清理**

1. 仅在绿色之后：消除重复、改进命名、提取辅助函数
2. 保持测试绿色，不添加新行为

**重复**

下一个失败测试，循环往复。

**输出结果：** 每个新函数/方法都有测试覆盖，验证清单全部通过，输出干净无错误。

## 5. writing-plans — 编写计划

### 功能描述

**主要作用：** 为多步骤任务编写全面的实现计划，假设执行者对我们代码库零上下文。文档化所有内容：每个任务要操作的文件、代码、测试、参考文档、验证方式。

**适用场景：**
- 拥有规格说明或需求后、动手编码之前
- 多步骤任务需要结构化的实施指导
- brainstorming 流程的自然下一步

**核心特性：**
- 零上下文假设：假设执行者了解开发但不知道工具或领域
- 粒度控制：每步 2-5 分钟，一口大小
- 完整内容要求：禁止占位符（TBD/TODO），每步包含实际代码
- 自审查机制：规格覆盖检查、占位符扫描、类型一致性检查
- 双执行选项：子代理驱动（推荐）和行内执行
- 标准计划结构：目标、架构、技术栈、任务分解

### 完整工作流程

**步骤 1：宣布开始**
- "I'm using the writing-plans skill to create the implementation plan."

**步骤 2：范围检查**
- 如果规格覆盖多个独立子系统，建议拆分为独立计划

**步骤 3：文件结构规划**
- 在定义任务前，先规划哪些文件创建或修改，每个文件的职责
- 设计清晰边界和良好接口的单元
- 共同变更的文件应放在一起
- 遵循现有代码库的惯例

**步骤 4：编写任务**
- 每个任务包含：创建/修改/测试的文件路径、具体步骤、实际代码、确切命令和预期输出
- 步骤格式：checkbox + 描述 + 代码/命令
- 禁止占位符：不用 TBD/TODO/implement later

**步骤 5：自审查**
- 规格覆盖：每个需求都有对应任务
- 占位符扫描：检查红牌标志
- 类型一致性：后续任务的类型/方法签名与前面定义一致

**步骤 6：保存计划**
- 保存到 `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`

**步骤 7：执行交接**
- "Plan complete. Two execution options: Subagent-Driven (recommended) or Inline Execution. Which approach?"

**输出结果：** 完整的 `YYYY-MM-DD-<feature-name>.md` 实现计划文档。

## 6. executing-plans — 执行计划

### 功能描述

**主要作用：** 加载计划、批判性审阅、执行所有任务、完成时报告。核心原则是严格按照计划步骤执行，不跳过验证。

**适用场景：**
- 已编写好的实现计划需要执行
- 在独立会话中按计划逐步实现（非子代理模式）

**核心特性：**
- 三步流程：加载审查 → 执行任务 → 完成开发
- 批判性审阅：执行前识别计划中的问题或疑虑
- 阻塞处理：遇到阻塞立即停止并提问，不猜测
- 验证不跳过：严格执行计划中指定的验证步骤
- 集成要求：必须配合 using-git-worktrees 确保隔离工作空间
- 完成后衔接：执行完毕自动调用 finishing-a-development-branch

### 完整工作流程

**Step 1：加载和审查计划**

1. 读取计划文件
2. 批判性审阅：识别任何问题或疑虑
3. 如有疑虑：在开始前向人工伙伴提出
4. 如无疑虑：创建 TodoWrite 并继续

**Step 2：执行任务**

对每个任务：
1. 标记为 in_progress
2. 严格按照每个步骤执行（计划已有 bite-sized steps）
3. 按照规定运行验证
4. 标记为 completed

**Step 3：完成开发**

完成所有任务并验证后：
- 宣布："I'm using the finishing-a-development-branch skill to complete this work."
- 调用 superpowers:finishing-a-development-branch 完成分支工作

**停止条件：**
- 遇到阻塞（缺失依赖、测试失败、指令不清）
- 计划有严重缺陷
- 不理解某个指令
- 验证反复失败

**输出结果：** 所有计划任务完成，验证通过，过渡到分支完成流程。

## 7. subagent-driven-development — 子代理驱动开发

### 功能描述

**主要作用：** 通过为每个任务派遣全新的子代理来执行计划，每个任务后经过两阶段审查：先规格合规审查，再代码质量审查。核心原则是"新子代理 + 两阶段审查 = 高质量、快速迭代"。

**适用场景：**
- 有实现计划且任务相对独立
- 在同一会话中执行（不需要上下文切换）
- 需要高质量保证的执行场景

**核心特性：**
- 每个任务分派独立的实现子代理（隔离上下文）
- 两阶段审查：规格合规审查（代码是否匹配规格）→ 代码质量审查（实现是否良好）
- 模型选择策略：机械任务用便宜模型，集成判断用标准模型，架构设计用最强模型
- 子代理状态处理：DONE/DONE_WITH_CONCERNS/NEEDS_CONTEXT/BLOCKED 四种状态
- 连续执行：任务间不暂停，直到所有完成或遇到阻塞
- 最终审查：所有任务后做全量代码审查
- 集成要求：using-git-worktrees → writing-plans → requesting-code-review → finishing-a-development-branch

### 完整工作流程

**准备阶段：**

1. 一次性读取计划文件
2. 提取所有任务完整文本和上下文
3. 创建 TodoWrite 列出所有任务

**每个任务的执行循环：**

**1. 派遣实现子代理**
- 使用 implementer-prompt.md 模板
- 控制者提供完整任务文本和场景上下文
- 子代理实现、测试、提交、自我审查

**2. 处理子代理问题**
- 子代理提问 → 回答并提供上下文 → 重新派遣
- 直到没有更多问题

**3. 派遣规格合规审查子代理**
- 使用 spec-reviewer-prompt.md 模板
- 确认代码与规格匹配，无遗漏、无多余

**4. 处理规格审查结果**
- 通过 → 进入代码质量审查
- 未通过 → 子代理修复 → 重新审查

**5. 派遣代码质量审查子代理**
- 使用 code-quality-reviewer-prompt.md 模板
- 评估代码是否构建良好

**6. 处理质量审查结果**
- 通过 → 标记任务完成
- 未通过 → 子代理修复 → 重新审查

**7. 处理下一任务**
- 循环直到所有任务完成

**最终阶段：**
- 派遣最终代码审查子代理审查整个实现
- 调用 superpowers:finishing-a-development-branch 完成

**模型选择指南：**
- 1-2 文件+完整规格 → 便宜模型
- 多文件+集成问题 → 标准模型
- 设计判断/全局理解 → 最强模型

**输出结果：** 所有任务经两阶段审查通过，代码提交，过渡到分支完成。

## 8. dispatching-parallel-agents — 派发并行代理

### 功能描述

**主要作用：** 当面临多个不相关的故障时（不同测试文件、不同子系统、不同 bug），并行调度代理分别调查。核心原则是"每个独立问题域派一个代理，让它们并发工作"。

**适用场景：**
- 3+ 测试文件因不同根因失败
- 多个子系统独立损坏
- 每个问题可以独立理解
- 调查之间无共享状态

**核心特性：**
- 四步模式：识别独立域 → 创建聚焦任务 → 并行派遣 → 审查整合
- 代理提示结构：集中、自包含、输出具体
- 约束设计：指定范围、明确目标、设定边界
- 审查整合：验证修复不冲突、运行全量测试套件
- 效果验证：3 个问题并行解决，时间为 1 个问题

### 完整工作流程

**步骤 1：识别独立问题域**

- 按"什么坏了"对故障进行分组
- 确认每个域独立 —— 修复一个不影响其他

**步骤 2：创建聚焦的代理任务**

每个代理获得：
- **具体范围：** 一个测试文件或子系统
- **明确目标：** 让这些测试通过
- **约束：** 不改其他代码
- **期望输出：** 发现和修复了什么

**步骤 3：并行派遣**

```typescript
Task("Fix agent-tool-abort.test.ts failures")
Task("Fix batch-completion-behavior.test.ts failures")
Task("Fix tool-approval-race-conditions.test.ts failures")
```

**步骤 4：审查和整合**

代理返回后：
1. 阅读每个总结
2. 验证修复不冲突
3. 运行全量测试套件
4. 整合所有变更

**输出结果：** 多个问题并行修复，修复无冲突，全量测试通过。

## 9. verification-before-completion — 完成前验证

### 功能描述

**主要作用：** 强制要求在声明工作完成、修复或通过之前，运行验证命令并确认输出。核心原则是"先有证据，再有断言"。

**适用场景（强制）：**
- 任何成功/完成声明的变体之前
- 任何满意度表达之前
- 提交、创建 PR、任务完成之前
- 移动到下一个任务之前
- 委托给代理之前

**核心特性：**
- 铁律：没有新鲜验证证据，就不能声称完成
- 五步门禁函数：识别 → 运行 → 读取 → 验证 → 断言
- 常见失败映射表：测试、linter、构建、bug 修复等的验证要求
- 红牌机制：出现"should"、"probably"、"seems to"等词立即停止
- 防合理化表：覆盖 8 种跳过借口
- TDD 回归验证：写测试 → 运行通过 → 撤销修复 → 必须失败 → 恢复 → 通过
- 代理委派验证：不信任代理成功报告，检查 VCS diff

### 完整工作流程

在声明任何状态或表达满意度之前：

**步骤 1：识别**
- 哪个命令能证明这个声明？

**步骤 2：运行**
- 执行完整命令（新鲜运行，非之前结果）

**步骤 3：读取**
- 完整输出，检查退出码，计算失败数

**步骤 4：验证**
- 输出是否证实声明？
  - 否：用证据说明实际状态
  - 是：用证据支持声明

**步骤 5：断言**
- 只在验证通过后才做出声明

**验证要求映射：**

| 声明 | 要求 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令输出：0 失败 | 之前运行、"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | Linter 通过、日志看起来好 |
| Bug 修复 | 测试原始症状：通过 | 代码改了、假设修复 |
| 需求满足 | 逐项检查清单 | 测试通过 |

**输出结果：** 声明有验证证据支撑，无未验证的完成声明。

## 10. requesting-code-review — 请求代码审查

### 功能描述

**主要作用：** 派遣代码审查子代理来在问题扩散前捕获它们。审查者获得精确的评估上下文，而非会话历史，确保聚焦在产出物上。

**适用场景：**
- 子代理驱动开发中每个任务完成后（强制）
- 主要功能完成后（强制）
- 合并到主分支前（强制）
- 卡住需要新视角时（可选）
- 重构前基线检查（可选）

**核心特性：**
- 基于 git SHA 的精确定位：从 base 到 head 的 diff
- 专业审查模板：code-reviewer.md 提供结构化评估框架
- 严重度分级：Critical（立即修复）> Important（继续前修复）> Minor（稍后处理）
- 允许技术反驳：如果审查者错了，用技术推理推回
- 集成工作流：子代理驱动开发中每个任务后自动触发

### 完整工作流程

**步骤 1：获取 git SHA**

```bash
BASE_SHA=$(git rev-parse HEAD~1)
HEAD_SHA=$(git rev-parse HEAD)
```

**步骤 2：派遣代码审查子代理**

使用 Task 工具，填充 `code-reviewer.md` 模板：
- `{DESCRIPTION}` — 构建内容的简要总结
- `{PLAN_OR_REQUIREMENTS}` — 应该做什么
- `{BASE_SHA}` — 起始 commit
- `{HEAD_SHA}` — 结束 commit

**步骤 3：根据反馈行动**

- Critical 问题：立即修复
- Important 问题：继续前修复
- Minor 问题：稍后处理
- 审查者错误：用推理推回

**输出结果：** 审查报告包含优势、问题和评估（Ready/Needs work）。

## 11. receiving-code-review — 接收代码审查

### 功能描述

**主要作用：** 代码审查需要技术评估而非情绪表现。核心原则是"先验证再实现，先提问再假设，技术正确胜过社交舒适"。

**适用场景：**
- 收到任何代码审查反馈时
- 反馈不清晰或技术上可疑时
- 外部审查者提出建议时

**核心特性：**
- 六步响应模式：读取 → 理解 → 验证 → 评估 → 响应 → 实现
- 禁止表演性同意：禁止"You're absolutely right!"、"Great point!"等回复
- 不清晰反馈处理：只要任一项目不清晰，暂停全部，先澄清
- 外部审查五步验证：技术正确性、功能影响、当前实现原因、跨平台兼容、审查者上下文
- YAGNI 检查：审查者建议的功能如果无实际使用则反驳
- 正确反馈确认方式：用事实陈述替代感谢
- GitHub 集成：回复到行内评论线程

### 完整工作流程

收到代码审查反馈时：

**步骤 1：READ** — 完整阅读反馈，不做反应

**步骤 2：UNDERSTAND** — 用自己的话重述需求（或提问）

**步骤 3：VERIFY** — 对照代码库现实检查

**步骤 4：EVALUATE** — 对这个代码库技术上是否合理？

**步骤 5：RESPOND** — 技术确认或有理推回

**步骤 6：IMPLEMENT** — 一次一项，测试每项

**处理不同来源：**

**来自人工伙伴（可信）：**
- 理解后实现
- 范围不清晰仍提问
- 跳过表演性同意
- 直接行动或技术确认

**来自外部审查者：**
- 先做五步验证检查
- 怀疑时用技术推理推回
- 冲突人工伙伴决策时先讨论
- 信号语（不舒服推回时）："Strange things are afoot at the Circle K"

**不清晰反馈处理：**
- 任一项目不清晰？→ 暂停全部
- 先全部澄清，再按顺序实现：阻塞 > 简单修复 > 复杂修复

**输出结果：** 经过验证的反馈被实现（或被技术推理驳回），代码得到改进。

## 12. finishing-a-development-branch — 完成开发分支

### 功能描述

**主要作用：** 引导完成开发工作，呈现清晰的选项并处理选定的工作流。核心原则是"验证测试 → 检测环境 → 呈现选项 → 执行选择 → 清理"。

**适用场景：**
- 实现完成、所有测试通过时
- 需要决定如何整合工作（合并/PR/保留/丢弃）
- 执行计划或子代理驱动开发的最终步骤

**核心特性：**
- 六步流程：验证测试 → 检测环境 → 确定基分支 → 呈现选项 → 执行选择 → 清理
- 环境感知：自动检测普通仓库 / 命名分支工作树 / 分离 HEAD
- 四种标准选项：本地合并 / 推送创建 PR / 保持现状 / 丢弃
- 安全机制：丢弃需键入确认、合并不验证不执行
- 工作树清理：智能识别 Superpowers 创建的工作树 vs 外部管理的
- 自修复：`git worktree prune` 清理过期注册

### 完整工作流程

**Step 1：验证测试**

```bash
npm test / cargo test / pytest / go test ./...
```

- 失败 → 停止，不能继续
- 通过 → 继续 Step 2

**Step 2：检测环境**

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
```

三种状态：
- GIT_DIR == GIT_COMMON → 普通仓库（4 选项）
- GIT_DIR != GIT_COMMON，命名分支 → 工作树（4 选项）
- GIT_DIR != GIT_COMMON，分离 HEAD → 分离 HEAD（3 选项）

**Step 3：确定基分支**

```bash
git merge-base HEAD main 2>/dev/null || git merge-base HEAD master 2>/dev/null
```

**Step 4：呈现选项**

标准 4 选项：
1. 合并回基分支
2. 推送并创建 PR
3. 保持分支现状
4. 丢弃此工作

分离 HEAD 的 3 选项（无合并选项）

**Step 5：执行选择**

- **选项 1（合并）：** 切回主仓库 → checkout 基分支 → pull → merge → 验证测试 → 清理工作树 → 删除分支
- **选项 2（PR）：** push 分支 → `gh pr create` → 不清理工作树
- **选项 3（保留）：** 报告状态 → 不清理
- **选项 4（丢弃）：** 显示删除内容 → 要求键入 "discard" → 清理工作树 → `git branch -D`

**Step 6：清理工作空间**

- 仅选项 1 和 4 执行清理
- 工作树路径在 `.worktrees/`、`worktrees/` 或 `~/.config/superpowers/worktrees/` → Superpowers 创建的 → 清理
- 其他路径 → 外部管理 → 不清理

**输出结果：** 工作整合完成，测试验证通过，工作空间清理干净。

## 13. using-git-worktrees — 使用 Git 工作树

### 功能描述

**主要作用：** 确保工作在隔离的工作空间中进行。优先使用平台的本地工作树工具，回退到手动 git worktree。核心原则是"检测现有隔离 → 使用原生工具 → 回退到 git，不与工具对抗"。

**适用场景：**
- 启动需要与当前工作空间隔离的功能工作时
- 执行实现计划之前
- 需要独立分支环境时

**核心特性：**
- 五步流程：检测现有隔离 → 创建隔离空间 → 项目设置 → 验证基线
- 双机制：原生工作树工具（优先）vs Git 工作树回退
- 目录优先级：已有本地目录 > 已有全局目录 > 指令文件 > 默认 `.worktrees/`
- 安全验证：项目本地目录先验证被 gitignore 忽略
- 子模块保护：区分工作树和子模块（避免误判）
- 沙箱回退：权限错误时在当前目录工作
- 项目自动设置：检测 package.json / Cargo.toml / requirements.txt 等

### 完整工作流程

**Step 0：检测现有隔离**

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
BRANCH=$(git branch --show-current)
```

子模块保护：
```bash
git rev-parse --show-superproject-working-tree 2>/dev/null
```

- GIT_DIR != GIT_COMMON（且非子模块）→ 已在工作树中，跳到 Step 3
- GIT_DIR == GIT_COMMON（或子模块）→ 需要创建工作树
- 询问用户同意

**Step 1a：原生工作树工具（优先）**

- 检查是否有 EnterWorktree、WorktreeCreate、`/worktree` 命令等
- 有则使用，跳到 Step 3

**Step 1b：Git 工作树回退（无原生工具时）**

**目录选择优先级：**
1. 用户指令文件中的声明
2. 已有 `.worktrees`（优先）或 `worktrees`
3. 已有 `~/.config/superpowers/worktrees/<project>`
4. 默认 `.worktrees/`

**安全验证：**
```bash
git check-ignore -q .worktrees 2>/dev/null
```
未忽略 → 添加到 .gitignore 并提交

**创建工作树：**
```bash
git worktree add "$path" -b "$BRANCH_NAME"
cd "$path"
```

**Step 3：项目设置**

自动检测并运行：
```bash
[ -f package.json ] && npm install
[ -f Cargo.toml ] && cargo build
[ -f requirements.txt ] && pip install -r requirements.txt
[ -f go.mod ] && go mod download
```

**Step 4：验证干净基线**

```bash
npm test / cargo test / pytest / go test ./...
```

- 失败 → 报告并询问是否继续
- 通过 → 报告就绪

**输出结果：** 隔离工作空间就绪，测试通过，准备实现功能。

## 14. writing-skills — 编写技能

### 功能描述

**主要作用：** 将 TDD（测试驱动开发）应用于流程文档编写。核心原则是"如果你没有看到代理在没有技能时失败，你就不知道技能是否教会了正确的东西"。

**适用场景：**
- 创建新技能时
- 编辑现有技能时
- 部署前验证技能是否有效时

**核心特性：**
- TDD 映射：测试用例 = 子代理压力场景，生产代码 = SKILL.md
- RED-GREEN-REFACTOR 应用于文档：基线（无技能）→ 编写技能 → 堵漏洞
- 四种技能类型：纪律执行型 / 技术型 / 模式型 / 参考型
- 防合理化机制：关闭每个漏洞、处理"精神 vs 字母"论证、构建合理化表、红牌列表
- CSO（Claude 搜索优化）：丰富描述字段、关键词覆盖、描述性命名、令牌效率
- 描述规则：只描述触发条件，不总结工作流（防止跳过阅读完整技能）
- 部署清单：每个技能 RED → GREEN → REFACTOR → 质量检查 → 部署 的强制步骤

### 完整工作流程

**RED 阶段 — 编写失败测试（基线）**

1. 创建压力场景（纪律技能需 3+ 组合压力）
2. 不用技能运行场景 —— 逐字记录基线行为
3. 识别代理使用的合理化模式

**GREEN 阶段 — 编写最小技能**

1. 名称只用字母、数字、连字符
2. YAML frontmatter：name + description（max 1024 chars）
3. Description 以"Use when..."开头，只描述触发条件
4. 第三人称描述
5. 全文覆盖关键词（错误、症状、工具）
6. 清晰的概述和核心原则
7. 解决基线测试中发现的具体失败
8. 代码内联或链接到独立文件
9. 一个优秀示例（不多语言）
10. 带技能运行场景 —— 验证代理现在遵守规则

**REFACTOR 阶段 — 堵漏洞**

1. 识别测试中的新合理化
2. 添加明确的反驳（纪律技能）
3. 从所有测试迭代构建合理化表
4. 创建红牌列表
5. 重新测试直到无懈可击

**质量检查：**
- 仅在决策不明显时使用小流程图
- 包含快速参考表
- 包含常见错误部分
- 无叙事性故事
- 支持文件仅用于工具或大量参考

**部署：**
- 提交到 git 并 push（如已配置）
- 考虑贡献回主仓库（如有普适价值）

**输出结果：** 经过 TDD 循环验证的技能文档，包含测试证据。
